# state/ 写作规范

`state/` 的文档描述"系统现在是什么样"，面向人，必须与代码一致。eo-doc-manager 的 `sync` 和 `re-sync` 生成 `state/` 时严格遵守本规范。

> 文件名历史：原为 `doc/`，重构后改名为 `state/`（语义更准：state = 系统当前状态）。本规范文件沿用 `doc-style.md` 名字不做迁移，内容指代 `state/`。

## 核心原则

1. **事实而非规划**：只描述已实现的功能，不写"计划支持"、"未来扩展"——规划/设计属于**项目管理侧** `<project_root>/docs/`，不在 eo-doc/
2. **业务语言而非代码语言**：用产品/策划/运营能看懂的词汇描述系统行为
3. **代码是唯一真相来源**：规则、数值、时序、边界条件一律以源码为准；旧文档只作为"业务背景"参考

## 粒度划分

**按业务模块划分，非技术模块。**

- ✅ 业务模块：产品/策划视角的功能单元（如"作物种植"、"运输任务"、"商店"、"建造升级"）
- ❌ 技术模块：代码层面的组织（如"TaskManager 类"、"数据持久化层"）

**子类型独立成篇**：子系统内存在显著不同的子类型时，每个子类型独立一篇。例：
- `farm-planting.md`（种植玩法）
- `farm-harvesting.md`（收获玩法）
- 而不是合在一个 `farm.md`

**生成前先枚举业务模块清单**，与用户确认后再动笔；粒度不对时宁可重新枚举也不要硬写。

## 统一结构骨架

每篇 `state/` 按以下章节组织（缺项可跳过，但顺序固定）：

```markdown
## 概述
一段话说明这个模块解决什么业务问题、在系统中的位置。

## 核心玩法 / 核心流程
玩家/用户视角的主要操作路径与典型场景。

## 规则与参数
触发条件、数值公式、时序、优先级、边界条件。引用配置表名+字段名作为规则锚点。

## 与其他模块的交互
输入（依赖哪些模块的产出）、输出（影响哪些模块）。

## 当前未实现 TODO
代码中 TODO/FIXME 注释涉及的业务功能。
```

## 写作风格 — 规格说明体

- **业务语言**：用"玩家"、"作物"、"任务"等业务词，不用"实例"、"回调"、"状态机"
- **允许引用配置锚点**：可写 `` `FarmConfig.GrowDuration` ``、`` `BuildingConfig.Level` `` 等
- **禁止贴代码**：不贴路径、类名、函数名、代码片段
- **时序用自然语言**：不用"OnUpdate 回调"、"Coroutine"，而写"每隔 N 秒检查一次"、"玩家点击后立即结算"

### 错误示例

```markdown
❌ TaskManager.OnTaskComplete() 被调用时，会通过 RewardService.Grant() 发放奖励
❌ 状态机进入 Running 状态后，执行 Update() 循环直到 Complete
❌ 调用 farmAgent.Harvest() 触发收获逻辑
```

### 正确示例

```markdown
✅ 任务完成时，系统向玩家背包发放 RewardConfig.Items 中配置的物品
✅ 作物进入成熟阶段后，玩家可点击收获；收获后作物格位进入待播种状态
✅ 收获产出数量 = FarmConfig.BaseYield × (1 + 品质加成)
```

## 不贴代码原则的边界

**允许**：
- 引用配置表名和字段名（作为业务规则的锚点）
- 引用协议消息名（如 "客户端发送 ReqPurchase 触发采购"）
- 引用枚举的业务值（如 "任务状态：待执行 / 执行中 / 已完成"）

**禁止**：
- 文件路径（如 `src/lib/task.cs`）
- 类名/函数名/函数签名
- 代码片段（任何 ``` 代码块）
- 目录结构 tree

## 生成时的工作顺序

1. **枚举业务模块**：从源码目录结构 + 配置表三方交叉验证，产出业务模块清单
2. **与用户确认清单**：模块划分不对就重来，不要硬写
3. **逐模块从源码提取规则**：
   - 读取模块对应的代码，提取业务规则、数值公式、时序、边界
   - 读取相关配置表，提取参数锚点
   - 读取 Proto 协议，识别对外交互点
4. **最后参考旧文档**：只从中提取"代码无法推断的业务背景描述"（为什么这样设计、历史决策原因）
5. **按统一结构骨架组织**：不要按代码结构组织

## 与 agent-handbook 的分工

| state/ | agent-handbook/ |
|------|-----------------|
| "系统做了什么" | "代码在哪里、怎么调用" |
| 业务语言 | 代码语言 |
| 玩家/策划看 | AI 看 |
| 不贴路径/类名 | 必须精确到文件和行号 |

同一个模块 state/ 和 agent-handbook/ 各一篇，交叉引用。
